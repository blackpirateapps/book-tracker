# Book Tracker Feature Documentation

This document was generated by analyzing the current codebase (`src/` + `api/`) and describes the features that are actually implemented in this project.

## Overview

This is a React + Vite book tracker web app with:

- A public-facing library view
- A password-gated admin dashboard
- Serverless API routes (`/api/*`) for reads/writes
- Turso (LibSQL) database storage
- OpenLibrary integration for adding books
- Vercel Blob + `sharp` for cover image storage/optimization

## Main User-Facing Features (Public App)

### 1. Public Home / Library Index (`/`)

Implemented in `src/pages/Home.jsx`.

- Book browsing with paginated loading (`limit=50`, `offset`)
- Text search across library (via `/api/public?q=...`)
- Shelf-based grouping in the UI:
  - `currentlyReading` ("Reading Now")
  - `read` ("Recently Finished")
  - `watchlist` ("To Read")
  - `abandoned`
- Shelf count summary links (anchor links to sections)
- Tag preview panel (first ~20 tags from global tag map)
- "Load More Entries" button for additional books
- Error state for failed list load
- Deduping of loaded books by `id`

### 2. Progressive List Enrichment

Also in `src/pages/Home.jsx`.

- Initial non-search list items are treated as partial objects (`_isPartial`)
- Background re-fetch enriches entries to full data after list render
- Improves perceived performance for the main feed

### 3. Book List Cards

Implemented in `src/components/BookListItem.jsx`.

- Small cover thumbnail
- Title and author display
- Tag badges (resolved from tag IDs)
- Reading progress badge for `currentlyReading`
- Defensive parsing for JSON-string authors/tags/image fields
- Partial-item placeholder state (`authors = "..."`)

### 4. Random Highlight / Quote of the Day

Implemented in `src/components/RandomHighlight.jsx` using `/api/public?action=randomHighlight`.

- Fetches a random highlight from books with `hasHighlights = 1`
- Displays quote text, author, and title
- Loading state
- Graceful empty state (returns null if no highlights)

### 5. Home Reading Activity Chart

Implemented in `src/components/HomeStats.jsx` using `recharts`.

- Fetches `/api/public?action=stats`
- Shows recent reading activity bar chart (last 5 years)
- Displays books-read-per-year summary chart on homepage sidebar

### 6. Book Details Page (`/book/:id`)

Implemented in `src/components/BookDetails.jsx`.

- Fetches full book details from `/api/books?id=<id>`
- Accepts `initialData` from navigation state to render immediately
- Robust JSON parsing for:
  - `authors`
  - `imageLinks`
  - `highlights`
  - `tags`
- Metadata panel:
  - authors
  - publisher
  - published date
  - pages
  - tags
- Reading log panel:
  - shelf/status
  - reading progress
  - medium
  - started date
  - finished date
- Description section
- Highlights section with count and quote-style rendering
- Back navigation

### 7. Statistics Page (`/stats`)

Implemented in `src/components/Stats.jsx`.

- Fetches aggregate stats from `/api/public?action=stats`
- Year filter dropdown (`all` + per-year selection)
- Summary cards for:
  - number of books
  - total pages
- Charts (Recharts):
  - books read per year
  - pages read per year
- Books grid for selected year/all years:
  - cover
  - title
  - author
  - page count
  - finish date
  - reading duration (derived from started/finished dates)
- Export screenshot/download of books grid as PNG (via `html2canvas`)
- Top authors table
- Reading medium distribution bars

## Admin / Dashboard Features (`/dashboard`)

Implemented primarily in `src/components/Dashboard.jsx`.

### 1. Password-Gated Dashboard Access (Client-side gate)

- Dashboard asks for password before showing admin tools
- Password is stored in component state and reused for write API calls
- Note: actual security enforcement happens in `/api/books` POST (server-side `ADMIN_PASSWORD` check)

### 2. Admin Book Library Table

- Fetch all books (`GET /api/books`)
- Refresh button
- Filter by title (client-side substring filter)
- Tabular view (title / author / shelf / actions)
- Actions:
  - `EDIT` (routes to `/dashboard/edit/:id`)
  - delete (`X`)

### 3. Add Book from OpenLibrary

Dashboard "Add Book" tool uses `src/components/Dashboard.jsx` + `/api/search` + `/api/books` POST.

- Search OpenLibrary (`/api/search?q=...`)
- Shows top matches
- Quick-add to shelf via buttons:
  - `W` => watchlist
  - `R` => currentlyReading
  - `F` => read

### 4. Export Hugo Payload

- Dashboard button calls `GET /api/books?action=export-hugo`
- Downloads JSON payload (currently saved as `hugo_highlights.md` in client code, but response is JSON)
- Export payload includes:
  - all read books (parsed JSON fields)
  - tags
  - flattened highlights
  - basic stats
  - generation timestamp

### 5. Upload / Parse Highlights (Admin Tool)

- Paste markdown content for highlights
- Select target book
- Preview highlights (client-side parser)
- Upload highlights to book (API-assisted parse + update)
- Option for replace/merge exists in state, but UI toggle is not currently rendered

Important implementation note:

- The dashboard currently calls `/api/books?action=parse-highlights` using a query param on a `POST`, but the API expects `POST /api/books` with body `{ action: "parse-highlights" }`.
- The API parser returns an object (`{ title, highlights }`), while dashboard upload flow currently treats it like an array.
- This means highlights upload is present in UI but likely needs a small bug fix to work end-to-end.

### 6. Edit Book Page (`/dashboard/edit/:id`)

Implemented in `src/pages/EditBookPage.jsx`.

- Loads full book by ID (`GET /api/books?id=...`)
- Admin password input (required to save)
- Editable fields include:
  - ID
  - title
  - authors
  - shelf
  - reading medium
  - reading progress
  - hasHighlights
  - started/finished dates
  - published/full publish dates
  - publisher
  - page count
  - description
  - subjects
  - tags
  - highlights
  - image links JSON
  - industry identifiers JSON
- Handles JSON validation for JSON textareas before save
- Sends partial/full update via `POST /api/books` (`action: "update"`)
- Success/error feedback messages

### 7. Tags Manager (Admin)

Implemented in `src/components/TagsManager.jsx`.

- Dashboard tab switch between `BOOKS` and `TAGS`
- Fetch tags (`/api/public?action=tags`)
- Fetch books list for bulk tag assignment (`/api/public?limit=1000`)
- Create tag (`tag-create`)
- Edit tag (`tag-update`)
- Delete tag (`tag-delete`)
- Bulk assign tag to selected books (`tag-bulk-add`)
- Simple color selection for tags (`blue`, `purple`, `green`, `emerald`, `amber`, `orange`, `red`, `gray`)

## API Backend Features (Implemented)

## `/api/public` (read-only public API)

Implemented in `api/public.js`.

- `GET /api/public`
  - List books (lightweight fields)
  - Pagination via `limit`, `offset`
  - Default sorting by shelf + date + title
- `GET /api/public?q=...`
  - Search across `title`, `authors`, `highlights`, `bookDescription`
- `GET /api/public?action=stats`
  - Returns reading stats for books on `shelf = 'read'`
  - Includes `booksByYear`, `authorStats`, `mediumStats`, totals
- `GET /api/public?action=randomHighlight`
  - Random highlight from books with `hasHighlights = 1`
- `GET /api/public?action=tags`
  - List all tags
- CORS enabled
- CDN caching headers (`s-maxage`, `stale-while-revalidate`)

## `/api/books` (admin + full book details API)

Implemented in `api/books.js`.

- DB initialization on request (creates tables/indexes if missing)
- `GET /api/books?id=...`
  - Full row for single book
- `GET /api/books`
  - Full list of all books (admin view)
- `GET /api/books?action=export-hugo`
  - Read-books export payload with parsed fields/highlights/tags/stats
- `POST /api/books`
  - Password-protected write actions (`ADMIN_PASSWORD`)
  - Supported actions:
    - `add`
    - `update`
    - `delete`
    - `parse-highlights`
    - `tag-create`
    - `tag-update`
    - `tag-delete`
    - `tag-bulk-add`

### `add` action capabilities

- Fetches OpenLibrary work metadata
- Fetches and uploads cover image to Vercel Blob
- Converts cover to optimized `.webp` via `sharp`
- Resolves author names via OpenLibrary author endpoints
- Inserts normalized row into `books` table

### `update` action capabilities

- Partial updates to any provided fields
- Supports changing primary `id` when `originalId` is provided
- Serializes object/array values to JSON strings for storage

### `parse-highlights` action capabilities

- Parses Markdown (`.md`) highlight files
- Parses Kindle-like HTML (`.html`) highlights
- Returns parsed `{ title, highlights }`

## `/api/search` (OpenLibrary search proxy)

Implemented in `api/search.js`.

- `GET /api/search?q=...`
- Proxies OpenLibrary search
- Returns trimmed result set (top 20)
- Caches responses

## How To Make / Set Up The API Backend

The project already includes the backend in `api/`. If you want to run it locally or recreate it for your own deployment, use the setup below.

### 1. Backend Architecture (Current)

- Runtime: Vercel Serverless Functions (Node, ESM)
- DB: Turso / LibSQL via `@libsql/client`
- Blob storage: Vercel Blob (`@vercel/blob`) for cover images
- Image processing: `sharp`
- External provider: OpenLibrary (search + work/author metadata)

### 2. Required Environment Variables

Add to `.env.local` (for local Vercel dev) and Vercel project settings:

- `TURSO_DATABASE_URL`
- `TURSO_AUTH_TOKEN`
- `ADMIN_PASSWORD`
- `BLOB_READ_WRITE_TOKEN` (required for cover upload in `add` action)

### 3. Local Development (API + Frontend)

- Install deps: `npm install`
- Install Vercel CLI if needed: `npm i -g vercel`
- Start local app with serverless emulation: `vercel dev`

Why `vercel dev`:

- Vite alone (`npm run dev`) runs the frontend but does not execute `api/*.js` serverless functions the same way as Vercel.
- `vercel dev` is the correct local environment for testing `/api/books`, `/api/public`, `/api/search`.

### 4. Minimum Backend Responsibilities (if rebuilding from scratch)

If you want to implement the same backend in another stack (Express/Fastify/Nest/etc.), keep these responsibilities:

- Public read API with pagination/search/stats/random highlight
- Admin write API with password validation
- Book CRUD operations
- Tag CRUD + bulk tag assignment
- OpenLibrary search + metadata import
- Highlight file parsing (Markdown and Kindle HTML)
- Cover image download + optimization + storage
- DB schema initialization/migrations

### 5. Recommended Improvements (Code-Analysis Notes)

These are not blockers for documentation, but they are worth fixing in backend/frontend integration:

- Fix dashboard highlight upload API call shape (`parse-highlights` action should be sent in POST body, not query string).
- Fix dashboard handling of parse response (`{ title, highlights }` vs array).
- `Export Hugo` client code should save as `.json` (response is JSON, not markdown).
- Consider moving repeated JSON parse/normalize logic into shared helpers.
- Add validation for shelf values and numeric ranges on server-side updates.
